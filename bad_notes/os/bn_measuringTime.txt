
Both rdtsc and clock_gettime are timing mechanisms in Linux/Unix systems, but they have significant differences in precision, overhead, and use cases.

_____________________________________________________________________________________________________

      SHORT NOTE:
        
        __rdtsc (IS AN UNPRIVILEGED CPU INSTRUCTION):
            Overhead: 20-30 cycles
            Precison: cycle level nanosecond
            rdtsc is portable ONLY on  x86/x86_64
            rdtsc does not generate system calls, interrupts, or preempt the CPU directly.
            (cpu premeption may ofc still be caused by OS, just not due to rdtsc itself.)
        
        clock_gettime (IS A SYSTEM CALL):
            Overhead: 100-1000 cycles
            Precison: depends on clock source
            portable across POSIX systems




_____________________________________________________________________________________________________


     <1> RDTSC (Read Time-Stamp Counter):

          rdtsc is a processor instruction that reads the Time Stamp Counter (TSC), which is a 64-bit register that counts CPU cycles since reset.
          Precision characteristics:

          Extremely high resolution (cycle-level precision)
          Typically in the nanosecond range on modern CPUs
          Raw access to CPU cycle counter

          Technical considerations:

          Direct CPU instruction (rdtsc) or the serializing variant (rdtscp)
          Very low overhead (few CPU cycles)
          Non-portable (x86/x86-64 specific)
          Not synchronized across cores/CPUs in older systems
          Subject to frequency scaling issues (counters may run at variable speeds with CPU frequency changes)
          Can be affected by CPU power states, turbo boost, and other frequency variations
          No system call overhead (user space instruction)

          ****************************************************************************************************************
            NOTE:
            The x86 architecture explicitly designates certain instructions as "unprivileged," meaning 
            they can be executed from any privilege level. The rdtsc instruction was deliberately designed by Intel/AMD
            to be accessible from all privilege levels, unlike many other hardware-accessing instructions.


            rdtsc DOES NOT BYPASS the kernel!! Entirely diff concept
          *****************************************************************************************************************

          Modern CPUs (post-2010) often have "invariant TSC" that runs at constant rate regardless of CPU frequency


    <2> clock_gettime():

          clock_gettime is a POSIX system call that provides access to various system clocks with different characteristics.
          Precision characteristics:

          Resolution depends on the clock source chosen
          Nanosecond precision available with proper clock sources
          More stable and consistent across system

          Technical considerations:

          System call overhead (higher than rdtsc)
          Portable across POSIX-compliant systems
          Different clock sources available:

          CLOCK_MONOTONIC: Monotonic time since some unspecified starting point
          CLOCK_MONOTONIC_RAW: Hardware-based time not subject to NTP adjustments
          CLOCK_REALTIME: Wall clock time (can jump if system time is changed)
          CLOCK_PROCESS_CPUTIME_ID: Per-process CPU time
          CLOCK_THREAD_CPUTIME_ID: Per-thread CPU time

          ******************************************************************************************************
                NOTE: 
                NTP is a networking protocol designed to synchronize clocks between computer systems
                over packet-switched, variable-latency data networks. It's one of the oldest Internet protocols
                still in use (developed in 1985).
          ******************************************************************************************************

          Synchronized across cores/CPUs
          Not affected by CPU frequency scaling or power states
          More stable for longer intervals
          Can be subject to NTP adjustments (depends on clock source)
