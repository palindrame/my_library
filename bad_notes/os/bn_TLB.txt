

The TLB accelerates the Virtual Address to Physical Address translation and IS PART OF THE MMU.
----------------------------------------------------------------------------------------------

          ❗❗IMPORTANT: The TLB stores PTE from << MULTIPLE >> Processes❗❗
          ❗❗It manages to do so using the ASID / PCID Bits❗❗
          
          ... more on this later.

TLBs are usually implemented as FULLY ASSOCIATIVE CACHEs to avoid CONFLICT misses in address translations.
                This simply means: Any Virtual address can be stored in any available slot in the TLB.

Modern processor designs often implement a multi-level TLB hierarchy:

     🔔 L1 TLB: Smallest but fastest, typically 64-128 entries  [💡The L1 TLB is often split into L1_iTLB and L1_dTLB]
     🔔 L2 TLB: Larger but slightly slower, often 512-2048 entries [💡The L2 TLB is generally unified, shared by instruction and data]

     For example:
      - Intel's recent x86-64 processors may have 64 entries in the L1 dTLB (data), 128 entries in the L1 iTLB (instruction), and 1536 entries in the L2 TLB
      - AMD's Zen architecture includes TLBs with similar capacities
      - ARM's recent designs might feature 48-128 entries for the L1 TLBs and 1024-2048 entries for the L2 TLB

      (For comparision a Page table for a process holds about 4,194,304 entries).


What is the xTLB ?:

                1. Instruction TLB [iTLB]:      VA to PA for CPU Instructions 
                2. Data TLB [dTLB]:             VA to PA for Program Data.

                Modern systems generally have both at the L1 TLB level.


Also a TLB Entry contains something that is not present in a Page Table Entry.
      A TLB does have Virtual Page Number, Physical Frame Number, Protection, Status bits BUT it ALSO has: 
      an ASID i.e. an Address Space IDentifier. This entry links each TLB entry to a process that is 
      running in the system.

          ❕Page Table Entry :     [ Page(VPN) | Frame(PFN) | Protection | Status ]              #key for lookup: VPN    #Associativity: Linear
          ❕TLB Entry :            [ Page(VPN) | Frame(PFN) | Protection | Status | ASID ]       #key for lookup: VPN    #Associativity: Fully or Set Associative

---------------------------------------------------------------------------------------------------------------------------------------------------------------------
                                ⌛  REMEMBER: In a particular Page Table, at a given time, all Entries belong to a single process, 
                                    but in a TLB, due to the ASID bits, we can have entries belonging to different processes simultaneously.
---------------------------------------------------------------------------------------------------------------------------------------------------------------------

     🚩  The ASID bits is what allows the TLB to store Entries from Multiple Processes SIMULTANEOUSLY; 
     🚩  Without it the TLB would have to be flushed and repopulated everytime a context switch occured,
     🚩  negating most of the benefits of having the TLB in the first place.

When does a TLB need to be Flushed?

The TLB is a cache, and like any cache, it can become inconsistent with the underlying page table. 
TLB flushing is required to maintain correctness.
TLB flush is needed in these scenarios:

              | Scenario                               | Reason                                                                          |
              | -------------------------------------- | ------------------------------------------------------------------------------- |
              | **Context Switch** (between processes) | Virtual addresses may refer to different physical pages in different processes. |
              | **Page Table Changes**                 | E.g., when a page is mapped/unmapped, permissions are changed.                  |
              | **TLB Shootdown** (in SMP systems)     | Page table is changed by one core but used by another.                          |
              | **Manual Flush Instruction**           | When OS explicitly wants to invalidate certain entries.                         |


TLB Optimization Using Entry Structure:
Modern processors optimize TLB flushing using structure-aware techniques, primarily involving:
a. ASID (Address Space Identifier):

    Each TLB entry is tagged with an ASID.
    During a context switch, instead of flushing the entire TLB, the hardware only considers entries with the current ASID.
    This enables TLB entries from other processes to coexist and be ignored without flushing.
    ✅ Result: Reduces the cost of a context switch dramatically.

b. Global Bit:

    Entries marked with the global bit are shared across all address spaces.
    Example: OS kernel pages.
    These entries are never flushed, even during context switches.

c. Selective Invalidation:

    Some CPUs allow partial TLB invalidation, e.g., invlpg instruction in x86 invalidates only the TLB entry for a given virtual address.

⁉️  Does Context Switching b/w threads need TLB flush?
        Yes, context switching between threads typically requires a TLB (Translation Lookaside Buffer) flush, though the extent depends on several factors.
        When switching between threads within the same process, a complete TLB flush is often unnecessary 
        since these threads share the same address space and page tables. 
        The virtual-to-physical mappings remain valid across the switch.
        However, when switching between threads of different processes, a TLB flush is generally required because:

                  1. Different processes use different page tables with different virtual-to-physical memory mappings
                  2. Without flushing, the TLB would contain stale translations that could lead to incorrect memory accesses
