
Virtual Address(VA) is an OS level abstraction. OS deals with VA pages.
                    Even Frames are an OS level abstraction of the actual hardware level Physical Address.

Why we need VA?
  ->Nice abstraction, prog needn't think about overstepping or overwriting critical memory.
  ->Therefore security as well.
  ->Prevents Fragmentation as well.
  ->Direct Memroy Access requires entire programs to be loaded into RAM before their execution starts,
    This might result in large programs being unable to run due to insufficient memory.
    Virtual Memory Allows us to use DEMAND PAGING. (see bn_demandPaging.txt)
  ->Every process has a common abstraction. VA starts with 0 and goes upto 2^48 (for 64 bit systems.)
  ->Can create shared pages (a single Physical Frame can be mapped to various Virtual Address Pages) for different processes to access.

The CPU deals only with VA. The OS is aware of VA and PA. 

As far as a process is concerned it only "sees" its own "contiguous "Virtual Address space, it is not bothered by any
Page table whatsoever. Page Table is an OS level construct, that processes are not aware of.

--------------------------------------------------------------------------------------------------------------------

📌 A question might arise: Why kernel space exists in virtual memory even though user processes can't access it ? 📌 

The inclusion of kernel space in the virtual memory map serves several critical purposes, even though it's inaccessible to user processes:

1. Address Translation Efficiency

    Single Page Table Structure:
    Having both user and kernel space in the same virtual address space allows the CPU to use a single set of page tables for both;
    During a context switch, only user space mappings need to change while kernel mappings remain constant
    This eliminates the need to flush the entire TLB when switching between user and kernel mode

2. System Call Performance

    Fast Transitions:
    When a system call occurs, the processor just needs to change privilege levels but doesn't need to switch page tables
    The kernel code and data are already mapped and ready for immediate access
    This significantly reduces system call overhead

3. Memory Management Practicality

    Direct Memory Access:
    The kernel needs to access both its own data structures and user space memory
    Having both in the same address space allows the kernel to directly reference user memory during system calls without complex remapping

4. Hardware-Level Protection

    Privilege Rings:
    The separation between user and kernel space is enforced by the CPU's protection rings
    This hardware-level enforcement is more efficient than maintaining completely separate memory spaces

5. Resource Utilization

    Shared Physical Memory:
    Both user and kernel virtual addresses ultimately map to the same physical memory
    This unification allows for flexible and dynamic allocation of physical memory resources

The MMU (Memory Management Unit) maps VA pages -> PA frames (both OS level constructs).

🔎 Virtually Indexed Caches:
   ------------------------
    🔹 Definition:
          A virtually indexed cache uses virtual address bits to index into the cache — before the TLB translates to a physical address.

    🔹 Step-by-Step Process:
        CPU generates a virtual address (VA).
        Index bits are directly extracted from the VA.
        Cache lookup begins in parallel with TLB lookup.
        Once TLB translation is done:
            The tag check uses the PA's tag bits.
            If the index bits were from the VA, they must match the PA bits to ensure correctness (see below).

    🔹 Conditions for Correctness:
         To avoid synonym/alias problems (same physical page mapped to different VAs), the cache must be small enough so that:
         Index bits come only from the page offset, which is the same in both VA and PA.
         So, Virtually Indexed, Physically Tagged (VIPT) cache requires:
         Index bits ⊆ page offset bits.
         Ensures same set is accessed even before translation is complete.

    🔹 Example:
         Assume:
             Page size = 4 KB → offset = 12 bits
             Cache set index uses 8 bits → fits in the 12-bit offset
             Can start cache access using these bits from VA

    🔹 Pros:
         Faster: Cache lookup and TLB translation can proceed in parallel.

    🔹 Cons:
         Must ensure no aliasing.
         Limited by page size (cache size and associativity must be small enough).
         More complex to design and manage.

🔎 Physically Indexed Caches:
   -------------------------
    🔹 Definition:
        A physically indexed cache uses physical address bits to index into the cache and locate the cache set.

    🔹 Step-by-Step Process:
        CPU generates a virtual address (VA).
        VA is sent to the TLB to obtain the physical address (PA).
        The index bits are extracted from the PA (e.g., middle bits of PA depending on cache size and block size).
        The PA tag and index are used to access the cache.

    🔹 Example:
        Assume:
          Cache size: 64 KB
          Block size: 64 B
          Cache associativity: 4-way

        Then:
          Number of sets = 64 KB / (4 × 64 B) = 256 sets
          Index bits needed = log₂(256) = 8 bits
        Those 8 bits are extracted from PA.

    🔹 Pros:
        No aliasing issues (same physical page accessed by two VAs).
        Simple coherence with memory.

    🔹 Cons:
        Slower access: Cache lookup must wait for TLB to complete translation.

When an instruction requires Memory Access, the processor forwards the VA to the MMU.
The MMU splits the address into two parts:  1. A (Virtual) page Number
                                            2. A page Offset

            This is VA ---->       [Page Number | Page Offset]
                                        MSBs          LSBs

The Page Number is the Sequential Number of page in the memory, and is represented by the MSBs of the VA.
The Page Offset is the location of a Single Byte of Memory within the Page and is represented by the LSBs of the VA.

🎌 How to go about thinking of this? Let's take an example:

   Suppose the CPU generates an N bit address. What do we do from there?

      1. First we need to know what the page size of our system is? Suppose the page size is 4KB = 4096B = 2^12B.
      2. Now we need a way to identify each byte of the page of size 2^12B, which is what the offset bits do,
         then we need 12 bits to represent the offset.
      3. Which effectively mean, now we have [N-12] bits to represent the Page Number bits.

      _________________________________________________________________________________________________________________________________


      4. To transalate a VA to a PA, the MMU extracts the Virtal Page Number (VPN) from the address generated by the processor
         and uses it to find the corresponding TLB entry to find the Physical Frame number(PFN) associated with this VPN.

                  4.a. If the Virtual Page Number is found in the TLB(i.e., TLB hit) and is marked as valid (the entry has a valid physical frame number),
                       the MMU replaces the (Virtual) page number with the corresponding (Physical) frame number and DOES NOT change the offset.
                       If the entry is found to be INVALID (cheking valid / invalid bit), the MMU will generate a page fault.

                          i.e.,

                          [ (Virtual)Page Number | Page Offset]        ---------MMU-------->       [ (Physical) Frame Number | Page Offset]                                           
                                     MSBs              LSBs                                                    MSBs                LSBs

                                        VIRTUAL ADDRESS                                                            PHYSICAL ADDRESS
                                                                                
                                                                  reminder: offset remains unchanged

                       📢NOTE📢: A TLB MISS DOES NOT generate a page fault, it results in us going over and checking in MM.
                                 A TLB hit and then a subsequent INVALID Frame no, causes the PAGE FAULT. 

                  _________________________________________________________________________________________________________________________________
                  

                  4.b. If the Virtual Page Number is not found in the TLB, The MMU sends a request to the Main Memory 
                    to look for the entry (by VPN lookup) in the full page table of the process.

                    📢NOTE📢
                    Page tables don't have fixed, predetermined physical addresses in RAM. They are:
                        2. Dynamically Allocated: The kernel allocates physical memory frames for page tables just like it does for other kernel data structures.
                        3. Kernel-Managed Memory: Page tables reside in memory that's managed by the kernel, not in user space.
                        4. Physically Contiguous: Each individual page table (but not the entire multi-level structure) is typically stored in physically contiguous memory.

                    Once the entry(lookup by VPN) is found in the Page Table (in Kernel-Managed Memory), it is checked for validity. 
                                <<<(This Page Table entry is also copied into the TLB for fast future references.)>>>
                    A valid (Physical)Frame Number is combined withthe offset from the original Virtual Address to create a complete physical address.

                          [ (Virtual)Page Number | Page Offset]        ---------MMU-------->       [ (Physical) Frame Number | Page Offset]                                           
                                     MSBs              LSBs                                                    MSBs                LSBs

                                        VIRTUAL ADDRESS                                                            PHYSICAL ADDRESS

                                                                  reminder: offset remains unchanged


      5. This Complete Physical Address is be used to access the Main Memory and retrieve the requested data using the Memory Controller.
      6. If the (Physical) Frame Number comes out to be Invalid, The MMU will generate AN EXCEPTION called "PAGE FAULT" and dispatch it to the OS.
         The same PAGE FAULT EXEPTION is generated, if the entry is found in the TLB, but is marked as INVALID.


      
      When the OS takes over, it performs checks to ensure that the process isn't trying to access memory it wasn't allocated.
      A violation often known as SEGMENTATION FAULT.
      If everything is okay: 
            1. the OS finds a Free Frame in Physical Memory,
            2. loads the requested page from disk,
            3. updates the Page Table Entry with the newly mapped Physical Address,
            4. finally, the accessed Page Table Entry is copied into the TLB for faster future lookups.

              🚨 🚨 BUT what happens if the TLB is full and can't add new entries ? 🚨 🚨  
                    
                    In this case the TLB CONTROLLER will evict an entry from
                    the TLB; and replace it with a new entry from the Page Table.

                    The Decision of which Entry to replace depends
                    on the TLB replacement policy.
                            - Some form of LRU
                            - Round Robin
                            - Random Replacement
