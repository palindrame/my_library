====================================================================================
                              DMA CONTROLLER (DMAC)
====================================================================================

Overview:
---------
DMA (Direct Memory Access) is a mechanism that allows hardware peripherals to
directly read from or write to main memory **without continuous CPU intervention**.
A **DMA Controller (DMAC)** orchestrates this data transfer independently of the CPU,
thus offloading memory-bound I/O operations and improving system performance.

====================================================================================
WHY DMA IS NEEDED – CPU UNBURDENING:
-------------------------------------
Before DMA, the CPU had to:
  - Poll or handle interrupts for every byte/word of data.
  - Read/write data from I/O ports or buffers.
  - Store that data into main memory manually.

This led to:
  ✗ CPU cycles being wasted in slow I/O-bound operations. [CPU busy doing not what it is primarily
    meant to do]
  ✗ Poor performance for high-throughput devices (e.g., disks, NICs). [CPU might be busy
    doing something else] 

💡Thus DMA is a boon for BOTH the CPU AS WELL AS high-throughput Devices.

DMA was introduced to:
  ✓ Allow devices to transfer data to/from memory autonomously.
  ✓ Let the CPU focus on computation or OS tasks during transfer.

====================================================================================
DMAC OPERATION MODES:
----------------------

1. **Burst Mode**:  -----------> extremely disruptive for cpu
   - DMAC takes full control of memory bus until entire transfer is complete.
   - High throughput, but CPU is blocked during transfer.

2. **Cycle Stealing**:  -------> Less disruptive for cpu than Burst mode
   - DMAC steals bus cycles between CPU operations.
   - CPU and DMAC share memory access → balanced performance.

3. **Transparent Mode**: ---------> most cpu friendly
   - DMA transfers occur only when CPU is idle.
   - No CPU performance impact, but slowest transfer.
     Can be resolved if the CPU has high Cache HITS,
     thus eliminating its need for frequent memory visits;
     and hence freeing up the memory bus for DMACs.

4. **Block Transfer Mode**:
   - Whole block of data is transferred per request.

====================================================================================
DMAC + CPU CACHING CONCERNS:
----------------------------
- CPU caches may become **incoherent** with memory after DMA writes.
- Solutions:
  ✓ Use cache flush/invalidate instructions (e.g., `CLFLUSH`).
  ✓ Use **uncacheable memory** regions for DMA buffers.
  ✓ Modern systems use **IO-MMUs** (e.g., Intel VT-d) to isolate and control DMA.

====================================================================================
DMA TYPES:
----------

→ **First Party DMA**:
   - Device (e.g., GPU, NIC) has internal DMA engine.
   - Initiates its own memory transfers.
   - Used in modern devices (PCIe).

→ **Third Party DMA**:
   - Central DMAC acts on behalf of peripheral devices.
   - Devices signal the DMAC (e.g., via IRQ or bus request line).
   - Common in older microprocessor systems (e.g., 8257 DMAC).

====================================================================================
CENTRAL DMA CONTROLLER (e.g., 8237 / 8257):
-------------------------------------------
- Interfaces between CPU, memory, and multiple peripherals.
- Has:
  ✓ 4 DMA channels.
  ✓ Address & count registers per channel.
  ✓ Priority resolution logic.
  ✓ Bus request/grant mechanism.

**Bus Arbiter**:
- Resolves conflicts between DMA and CPU or among multiple DMA channels.(known as Bus Contention)
- Implements:
  ✓ Fixed Priority
  ✓ Rotating Priority
  ✓ Daisy-chaining
          |
          |=================>   DAISY CHAINING (PRIORITY RESOLUTION)
          |                    ----------------------------------------

                                Concept:
                                --------
                                Daisy chaining is a **hardware-based priority resolution scheme** for multiple
                                DMA-capable devices requesting bus access through a single DMA controller.

                                Diagram:
                                --------
                                  +------+    +------+    +------+    +------+
                                  | Dev1 |--->| Dev2 |--->| Dev3 |--->| Dev4 |
                                  +------+    +------+    +------+    +------+
                                    |           |           |           |
                                    +-----------+-----------+-----------+--> DMAC (Bus Request)

                                Mechanism:
                                ----------
                                - Devices are physically connected in series.
                                - Bus Grant (BG) signal propagates through the chain.
                                - The **first device** in the chain with a Bus Request (BR) **captures the grant**.
                                - Devices later in the chain only get BG if earlier devices don't need it.

                                Priority:
                                ---------
                                ✓ Fixed, physical order.
                                ✓ Device closest to DMAC has **highest priority**.

                                Advantages:
                                -----------
                                - Simple hardware, no complex logic needed.

                                Disadvantages:
                                --------------
                                - Inflexible priority.
                                - Latency for low-priority devices.
                                - Failure in one device can block others.



====================================================================================
DMA IN PCIE EXPRESS SYSTEMS:
----------------------------
- Devices on PCIe bus (e.g., NVMe, GPUs, NICs) include **integrated DMA engines**.
- Use **First Party DMA**.
- DMA engines initiate and control transactions across PCIe fabric.
- Leverage PCIe TLPs (Transaction Layer Packets) to:
  ✓ Read/Write directly into system RAM.
  ✓ Use **scatter-gather** tables for non-contiguous buffers.
           |
           |
           |     ================================================================================
           |=====>                        SCATTER-GATHER DMA TABLES
           |     ================================================================================
           |


            I. OVERVIEW
            -----------
            Traditional DMA requires memory to be **physically contiguous**.
            But most modern systems allocate buffers in **non-contiguous pages** due to:
              ✓ Virtual memory usage
              ✓ Physical memory fragmentation

            **Scatter-Gather DMA (SG-DMA)** solves this by enabling the DMA engine to
            transfer data between non-contiguous memory blocks as if they were one
            contiguous buffer.

            ================================================================================
            II. CONCEPT
            -----------

            → "Scatter" refers to writing data from a device to many locations in memory  
            → "Gather" refers to reading data from many memory locations to send to a device

            DMA controller uses a **table of descriptors**, where each entry tells it:
              - Where to read/write
              - How much data
              - Whether it's the last chunk

            ================================================================================
            III. TABLE STRUCTURE
            --------------------

            Each SG Table Entry (aka **DMA Descriptor**) typically includes:

            +----------------+-------------------+-----------------+----------------------+
            | Buffer Address | Transfer Length   | Flags/Status    | Next Descriptor Ptr  |
            +----------------+-------------------+-----------------+----------------------+
            |  0x0012F000    |  4096 bytes       | VALID, INT ON   |   0x8000D000         |
            |  0x00341000    |  4096 bytes       | VALID, NO INT   |   0x8000D020         |
            |  ...           |  ...              | ...             |   ...                |
            +----------------+-------------------+-----------------+----------------------+

          DMA Engine walks through the table and executes the transfers sequentially.

          ================================================================================
          IV. ADVANTAGES
          --------------

          ✓ Works with fragmented, non-contiguous memory  
          ✓ Eliminates need for bounce buffers  
          ✓ Efficient for large or dynamic data  
          ✓ Offloads CPU by batching memory transfers

          ================================================================================
          V. USAGE FLOW (Step-by-Step)
          -----------------------------

          1. **Driver Allocates Scatter List**:
            - OS allocates pages in physical memory.
            - Builds scatter-gather table (list of physical segments).

          2. **Driver Maps Table to DMA Address Space**:
            - Uses `dma_map_sg()` (in Linux) to map pages to DMA addresses.
            - May invoke IOMMU translation.

          3. **DMA Engine is Programmed**:
            - Passed base address of the SG table.
            - Starts transfer and walks the list.

          4. **DMA Completes All Descriptors**:
            - Optionally generates interrupt after each or at end.
            - May automatically re-arm (looping SG mode).

          ================================================================================
          VI. DMA WITH SG IN PCIE AND IOMMU SYSTEMS
          ------------------------------------------

          ✓ Devices (NICs, GPUs, NVMe) on PCIe support SG DMA natively.
          ✓ Descriptors may be:
            - In system memory (read via PCIe reads)
            - In device-local memory (managed internally)
          ✓ IOMMU (e.g., Intel VT-d) allows:
            - Safe, virtualized SG mappings
            - Per-device memory protection and translation

          ================================================================================
          VII. RISKS AND CONSIDERATIONS
          ------------------------------

          ✗ SG DMA requires devices that support **chained descriptor walking** ----> DMA controller (or the DMA-capable device itself) 
                                                                                      automatically traverses a linked list of DMA 
                                                                                      descriptors (or entries), each describing a memory segment to be transferred. 
          ✗ Incorrectly ordered entries or invalid flags may cause:
            - Data corruption
            - Partial transfer
          ✗ Cache coherency must be ensured:
            - DMA to cached memory regions can lead to stale/inconsistent data

          ================================================================================



**IOMMU**:
- Translates device DMA virtual addresses → physical addresses.
  using I/O page tables.
- Enforces access control (security).
- Examples:
  ✓ Intel VT-d
  ✓ AMD IOMMU

====================================================================================
VIRTUAL MEMORY + DMA:
---------------------
- Peripheral devices operate using **physical addresses**, not virtual addresses.
- If a device needs to perform DMA to a buffer in user space:
   → OS must:
     1. Pin the virtual memory (prevent swapping).
     2. Translate VA → PA.
     3. Program the DMA engine with the PA.
     4. Optionally use bounce buffers for fragmentation.
                          |
                          |=============> BOUNCE BUFFERS (MEMORY FRAGMENTATION SOLUTION)
                          |              ---------------------------------------------------

                                        Context:
                                        --------
                                        Peripherals may not handle:
                                          ✗ Virtual memory addresses
                                          ✗ Fragmented or non-contiguous physical memory

                                        Definition:
                                        -----------
                                        A **bounce buffer** is a temporary, physically-contiguous memory buffer allocated
                                        by the OS to serve as an intermediate staging area for DMA.

                                        How it works:
                                        -------------
                                        1. CPU copies user-space buffer → bounce buffer (for write-to-device)
                                        2. Device performs DMA using bounce buffer (contiguous PA).
                                        3. After DMA:
                                          ✓ Bounce buffer → user buffer copy (for device-to-memory transfer)

                                        Used When:
                                        ----------
                                        ✓ The DMA engine cannot:
                                          - Access high memory
                                          - Work with non-contiguous pages
                                          - Use virtual addresses

                                        Trade-offs:
                                        -----------
                                        ✓ Ensures compatibility  
                                        ✗ Performance penalty due to extra memory copy
_____________________________________________________________________________________________________________

Modern systems support:
  ✓ DMA Remapping via IOMMU → devices can use DMA with virtual addresses safely.

====================================================================================
SUMMARY:
--------
✓ DMA allows direct memory access for peripherals, bypassing CPU bottlenecks.
✓ Central DMACs were used in legacy systems; modern devices use on-chip DMA engines.
✓ CPU caching and DMA must be coordinated to ensure memory coherency.
✓ DMA is essential in PCIe and high-speed I/O scenarios.
✓ Virtual memory and IOMMUs enable secure and efficient DMA in virtualized systems.
