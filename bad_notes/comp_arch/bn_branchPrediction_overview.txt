                                                                    OLD ROUGH NOTE ON BRANCH PREDICTOR
                                                                    ##################################
                        The branch predictor is also a dedicated hardware component:

                        Location: Physically located in the CPU's front-end, between the instruction fetch unit and the decoder

                        Physical proximity: Sits very close to the instruction cache (I-cache) but is a separate component from it

                        Integration: NOT part of the MMU or memory controller

                        Implementation: Consists of:

                              1. Pattern History Tables (PHTs): SRAM arrays storing branch history
                              2. Branch Target Buffer (BTB): A specialized cache storing branch target addresses
                              3. Return Address Stack (RAS): A small hardware stack for function return prediction
                              4. Global/local history registers: Shift registers tracking branch patterns
                              5. Predictor logic: Circuitry implementing prediction algorithms


                        The branch predictor works closely with the instruction fetch unit to determine the next instruction address, 
                        before the current branch is resolved. Its outputs feed directly into the program counter logic and instruction fetch mechanisms.
                        In modern CPU die photos, we can typically see the branch predictor as a distinct block near the L1 instruction cache
                        and fetch unit in the front-end of the pipeline, while reservation stations appear as regular structures near their
                        respective execution units throughout the execution core.

===============================================================================
          BRANCH PREDICTION IN MODERN CPU CORES — LOW-LEVEL TECHNICAL NOTE
===============================================================================

OVERVIEW:
---------
Branch Prediction = Technique to guess the outcome of **control-flow** instructions
BEFORE they are executed to **keep the pipeline full** and avoid stalls.

Instructions Affected:
----------------------
- Conditional branches (e.g., `if`, `while`)
- Indirect branches (e.g., virtual calls, function pointers)
- Unconditional jumps/calls (target prediction still needed)

===============================================================================
| Is branch prediction hardware or software?  
===============================================================================

✅ **Branch prediction is a microarchitectural HARDWARE mechanism**

It operates invisibly to the software and is part of the instruction fetch path.

🔸 Software (e.g., compilers) can **assist** by aligning branches or reordering code,  
but cannot control the internal predictor logic.

===============================================================================

===============================================================================
1. BRANCH TARGET BUFFER (BTB)
===============================================================================

Stores the predicted **target address** of recently executed branches.

- Indexed by lower bits of the PC
- Stores: PC → target address + type (cond/uncond/call/ret)
- Acts as a **cache of branch targets**
- Multiple levels possible (L1 BTB, L2 BTB)

===============================================================================
2. PATTERN HISTORY TABLE (PHT)
===============================================================================

Predicts **branch direction** (taken or not taken) using 2-bit saturating counters.

- Indexed using local/global history
- Entry = 2-bit counter:
    00 → Strongly Not Taken
    01 → Weakly Not Taken
    10 → Weakly Taken
    11 → Strongly Taken

===============================================================================
3. GLOBAL HISTORY REGISTER (GHR)
===============================================================================

A shift register tracking **outcomes of recent branches** (Taken = 1, Not Taken = 0)

- Length: 8–64 bits typically
- Used to **index PHT** with higher correlation than static prediction

===============================================================================
4. LOCAL HISTORY TABLE (LHT)
===============================================================================

Tracks **branch-specific** history (rather than global).

- Indexed by branch PC
- Output is a bit pattern used to index another PHT

This enables **per-branch behavior learning**.

===============================================================================
5. RETURN ADDRESS STACK (RAS)
===============================================================================

Predicts return addresses for `RET` instructions.

- Pushes return address on `CALL`
- Pops address on `RET`
- Prevents misprediction of `RET` targets in recursive or nested calls

===============================================================================
6. INDIRECT BRANCH PREDICTOR
===============================================================================

Predicts **target of indirect branches**, e.g., virtual function calls.

- Uses hashed PC + history to access a **target cache**
- Stores multiple possible targets with usage statistics

===============================================================================
IMPLEMENTATION CHARACTERISTICS
===============================================================================

✅ **Pipelined** — prediction made early during Fetch to avoid stalls

✅ **Speculative** — prediction may be wrong → flushed pipeline

✅ **Highly tuned and opaque** — Modern CPUs use:
   - Tournament predictors (choose best among local/global)
   - TAGE (TAgged GEometric history predictor)
   - Loop predictors
   - Neural predictors (e.g., perceptron-style in research)

===============================================================================
SHARING OF PREDICTION RESOURCES
===============================================================================

🔹 **Per-core**:
   - BTB, GHR, PHT, LHT, RAS are generally **per-core** to reduce contention and ensure locality.

🔹 **May be shared**:
   - L2 uOp cache / Trace cache (some Intel CPUs)
   - Very high-level or statistical profiling data (rare in real-time)

Execution units (ALU, FPU, etc.) are **not shared** between branch predictors.

===============================================================================
PIPELINE EFFECT & MISSPREDICTION
===============================================================================

- Mispredicted branches cause:
   ❌ Flush of in-flight instructions
   ❌ Pipeline bubbles (lost cycles)
   ❌ Rewind to correct fetch PC

- Penalty = ~10–20 cycles in modern CPUs (worse with deeper pipelines)

===============================================================================
SUMMARY
===============================================================================

✅ Branch prediction is **hardware-based** and **per-core**  
✅ Involves complex logic: BTB, PHT, GHR, LHT, RAS, Indirect Predictors  
✅ Shared prediction logic is rare; mostly local to each core  
✅ Essential for high-performance OoO and superscalar designs  
✅ Mismanagement causes pipeline flushes and huge performance loss
