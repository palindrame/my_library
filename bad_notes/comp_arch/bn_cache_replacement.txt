
===============================
CACHE REPLACEMENT POLICIES
===============================

1. CACHE REPLACEMENT AND EVICTION
---------------------------------
* Cache: A small, fast memory that stores recently accessed data for quick retrieval.
* Eviction: The process of removing an existing cache line to make space for new data.
* Replacement: The decision of *which* cache line to evict when space is needed.
* Cause: Occurs when a cache miss happens and the cache is full (no free block).
                                                                         ------
2. CACHE REPLACEMENT POLICY                                                  |
---------------------------                                                  | 
* A rule/algorithm that decides *which* "BLOCK" to evict during replacement. |
* Goal: Maximize hit rate and reduce miss penalties.                         |
                                                                             V
                                                                ------------------------------------------
                                                                        WHAT IS A BLOCK IN CACHE?
                                                                ------------------------------------------

                                    * **Cache Block (aka Cache Line)**: The smallest unit of memory that can be stored or replaced in the cache.
                                    * Typical Size: 32 to 128 bytes (common = 64B). Determined by CPU architecture and cache design.
                                    * Why Blocks? Caches exploit **spatial locality** (if you access address X, you'll likely access X+1 soon).

                                    * Hardware-Defined:
                                      - Block size is fixed at system design time (e.g., 64B in Intel CPUs).
                                      - All memory transactions to/from cache happen at block granularity.

                                    * Relation to Replacement Policy:
                                      - Block size is **ORTHOGONAL** to the replacement policy.
                                      - Replacement policy operates at block level but DOES *NOT* define the block size.

                                    * Cache Block Layout (typical):
                                      - **Tag**: Part of memory address used to identify block.
                                      - **Data**: Actual bytes from memory.
                                      - **Valid/Dirty bits**: Status indicators for cache coherence.

  Examples of Replacement Policies:

  - LRU (Least Recently Used): Evicts the block that hasn't been used for the longest time.     // based on WHEN a block was accessed?, it DGAF about how often![temporal loc]
  - LFU (Least Frequently Used): Evicts the block with the lowest access frequency.             // based on how often a block was accessed, it DGAF about when! [NOT spatial ig]
  - FIFO (First-In First-Out): Evicts the block that was loaded earliest into the cache.
  - RR (Random Replacement): Evicts a random block from the cache.
  - MRU (Most Recently Used): Evicts the block accessed most recently.
  - ARC (Adaptive Replacement Cache): Balances recency and frequency using TWO LRU LISTS.          // to know more about ARC, check  bn_cache_ARC_algorithms.txt
    - DBL (Dynamic Belady-Like): Adaptively simulates Belady’s optimal policy (approximate). --------------------------
    - CAR (Clock with Adaptive Replacement): Uses clock algorithm + frequency.                                        |

                                                                                                                      |
                                  ---------------------------------------------------------                           |
                                  BELADY’S ALGORITHM – OPTIMAL BUT UNREALIZABLE POLICY      <-------------------------|
                                  ---------------------------------------------------------

                                  **Belady's Algorithm (Optimal Replacement Policy)**: Always evicts the block that WILL be accessed
                                                                                       FARTHEST in the FUTURE.
                                  * Also called: **MIN**, **OPT** or **clairvoyant replacement policy**.

                                  * Key Property: Yields the lowest possible cache miss rate for a given trace of memory accesses.
                                  * Limitation: Requires *perfect knowledge of the future*, so cannot be implemented in real hardware.

                                  * Usage:
                                    - Used as a **benchmark** to compare real-world policies.
                                    - Useful in simulations to evaluate policy efficiency.

                                  New Concepts:
                                    - **Clairvoyant Policy**: A policy that assumes future knowledge of memory accesses (not implementable).
                                    - **Trace**: A recorded sequence of memory accesses, often used in simulation studies.

    ==============================================
    EXAMPLES OF CACHES AND THEIR REPLACEMENT POLICIES
    ==============================================

    1. L1 CACHE (Level-1 Cache)
    ----------------------------
    * Description: Smallest and fastest cache, typically split into instruction (L1i) and data (L1d).
    * Size: 16KB–64KB per core.
    * Policy: **Pseudo-LRU (PLRU)** or **True LRU** (if associativity is small, e.g., 2-way or 4-way).
    * Reason: Low latency is critical; simpler logic helps.

    2. L2 CACHE (Level-2 Cache)
    ----------------------------
    * Description: Larger, slower than L1 but still private per core in most CPUs.
    * Size: 128KB–1MB.
    * Policy: **Pseudo-LRU**, sometimes **Adaptive**.
    * Reason: Needs balance between performance and metadata overhead.

    3. L3 CACHE (Last Level Cache / LLC)
    -------------------------------------
    * Description: Shared across all CPU cores; typically unified (holds both instructions and data).
    * Size: 2MB–64MB or more.
    * Policy: **Adaptive Replacement**, **Dynamic Insertion Policy (DIP)**, or **Tree-based PLRU**.
    * Intel CPUs (e.g., Skylake, Alder Lake): Use undocumented adaptive strategies, influenced by workload behavior.
    * AMD CPUs: Also employ adaptive or probabilistic policies.

    4. TLB (Translation Lookaside Buffer)
    -------------------------------------
    * Description: Specialized cache for virtual-to-physical memory translations.
    * Policy: Often **LRU** or **Random Replacement**.
    * Reason: Fast but limited; less critical locality than data/instruction caches.

    5. GPU CACHES (e.g., NVIDIA, AMD GPUs)
    ---------------------------------------
    * Description: Specialized caches for parallel processing units (e.g., L1/shared cache, texture cache).
    * Policy: **Random**, **Round-robin**, or **Stream-aware** (since workloads can be stream-like).
    * Reason: Emphasis on bandwidth over hit rate; streaming workloads dominate.

    6. SOFTWARE CACHES (e.g., Browser Cache, Web CDN)
    --------------------------------------------------
    * Description: Implemented in software for file/content storage.
    * Policy: **LRU**, **LFU**, **ARC**, or custom ML-driven policies.
    * Reason: Flexibility in memory and runtime characteristics.

===============================================
SPACE COMPLEXITY OF LRU AND LFU POLICIES
===============================================

Let:
- N = associativity of a cache set (e.g., N-way set associative cache)

1. FULL LRU (True LRU)
------------------------
* Description: Maintains a total ordering of blocks from Most Recently Used (MRU) to Least Recently Used (LRU).
* Space: **O(N log N)** — typically implemented with:
  - A linked list or array of N elements, or
  - A timestamp per block (log N bits for ordering)
* Each block needs a unique rank or timestamp => log N bits * N entries.

2. TREE-BASED PLRU (Pseudo-LRU)
-------------------------------
* Description: Uses a binary tree of bits to approximate LRU.
* Space: **O(N)** — requires **(N - 1)** bits for an N-way set.
  - Example: 8-way PLRU needs 7 bits.
* Much cheaper than full LRU, but accuracy suffers in some access patterns.

3. NOT RECENTLY USED (NRU)
--------------------------
* Description: Each block has a single reference bit; bits periodically cleared to allow approximate aging.
* Space: **O(N)** — 1 bit per block.
* Used in CLOCK algorithm variants and TLBs.

4. QUAD-AGE LRU (Aging-Based Approximation)
-------------------------------------------
* Description: Each block has 2-3 bits to encode a coarse "age bucket".
* Space: **O(N)** — 2–3 bits per block.
* Gives moderate recency tracking with minimal cost.

------------------------------
LFU (LEAST FREQUENTLY USED)
------------------------------
* Description: Each block maintains a counter of how often it was accessed.
* Space: **O(N log F)** — where F is the max frequency value.              // each counter needs enough bits to represent F, which comes out to be log F.
  - Each block has a counter (e.g., 6–8 bits typical).
  - Needs periodic **decay/reset** to prevent saturation and to remain responsive.
* Variants like **LFU with aging** or **Approximate LFU** (e.g., TinyLFU) aim to bound counter size and improve practicality.

-------------------------------------
COMPARISON TABLE
-------------------------------------

| Policy          | Space Complexity | Notes                              |
|-----------------|------------------|------------------------------------|
| Full LRU        | O(N log N)       | Precise but expensive              |
| Tree-PLRU       | O(N)             | Approximate, very efficient        |
| NRU             | O(N)             | Coarse aging (1-bit per block)     |
| Quad-Age LRU    | O(N)             | Buckets of age, moderate accuracy  |
| LFU             | O(N log F)       | Needs decay/reset for long runs    |

3. DEEP DIVE: LRU (LEAST RECENTLY USED)    // specially designed to exploit temporal locality
---------------------------------------
* Tracks temporal locality by assuming recently used blocks will be used again soon.
* High hit rate for workloads with good recency patterns.
* Challenge: High implementation cost in hardware for full LRU on large associativity.

  A. FULL LRU (True LRU)
  ----------------------
  - Maintains a strict access ordering (e.g., by a stack or list).
  - On each access, the block is moved to the most-recent position.
  - Eviction removes the least-recently used (bottom of the stack).
  - Overhead: Requires O(N) metadata for N-way set associativity.    

  B. TREE-BASED PSEUDO LRU (PLRU)
  -------------------------------
  - Binary tree tracks usage with one bit per internal node.
  - Each bit tells which subtree was used more recently.
  - Navigate to LRU block by reading bits top-down.
  - On access, update bits along the path from leaf to root.
  - Approximation: Not exact LRU but cheaper (log₂N bits).

  C. NOT RECENTLY USED (NRU)
  --------------------------
  - Uses a reference bit per block (1 = accessed, 0 = not).
  - Periodically clears all bits to start a new epoch.
  - On eviction, prefers a block with ref bit = 0.
  - Simple to implement with low hardware cost.
  - Approximation of LRU with periodic reset intervals.

  D. QUAD-AGE LRU
  ---------------
  - Classifies cache blocks into 4 age bins (recent → old).
  - Each access promotes block to a younger bin.
  - Eviction removes block from the oldest non-empty bin.
  - Trade-off: Balances tracking accuracy and metadata size.
  - Good for approximate recency with constant-time operations.

SPECIALIZED CONTROL HARDWARE FOR CACHE REPLACEMENT
---------------------------------------------------

* Modern CPUs have dedicated **Cache Management Logic**: A HARDWARE component that tracks
  usage/access history and makes real-time cache replacement decisions.

* Purpose: Optimize latency and power by deciding "which block to evict" when cache misses occur.

* Cache replacement is implemented per cache level:
  - L1: Fastest, smallest; often uses simpler policies like **Pseudo-LRU**.
  - L2/L3: Larger, more associative; can use **LRU**, **Adaptive** or **Randomized** policies.

* Key Hardware Structures:
  - **LRU Stack or Counters**: For full or approximated LRU tracking (O(N) per set for N-way associativity).
  - **Reference Bits**: Single bits to track usage (used in NRU or Clock-based algorithms).
  - **Tree-Based Selectors**: Binary trees to encode "less-recently used" paths in PLRU.
  - **State Machines**: Control logic that uses policy-specific state transitions to choose victims.

* Example: Intel CPUs
  - Use proprietary adaptive policies depending on workload type and cache level.
  - LRU is often *approximated* using metadata bits per line or tree-based PLRU.
  - **Dynamic Insertion Policy (DIP)**: A runtime method to switch between LRU and BIP (see below).

  New Concepts:
  - DIP (Dynamic Insertion Policy): Uses sets of cache lines as policy testers to adapt between
                                    LRU and BIP(Bimodal Insertion Policy).
  - BIP (Bimodal Insertion Policy): Inserts most lines at LRU position, with occasional insertion
    at MRU (reduces pollution from streaming data).
