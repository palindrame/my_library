                   "The unit of transfer between the memory and cache is called a block or a line.

   🔭 The info a cache line holds is dependent of the cache organization.

SECTION 1: READ vs WRITE POLICIES
──────────────────────────────────────────────────────────────────────────────
- READ POLICY: Dictates what happens when the CPU requests data from memory.
  - Read Hit: Data is found in cache → serve directly.
  - Read Miss: Data not in cache → load from lower memory (e.g., RAM) into cache.

- WRITE POLICY: Dictates what happens when the CPU writes/updates data.
  - Write operations are more complex due to:
    → Data consistency needs (e.g., cache vs main memory).
    → Possible data sharing between cores (leads to coherence issues).
    → Writes may go in both directions (cache & memory).

→ Writes are generally slower because:
  - Cache coherence maintenance is required (more on this later).
  - Dirty data may need to be flushed (in write-back).
  - Writes may go to multiple locations (cache, memory, and other cores).

──────────────────────────────────────────────────────────────────────────────
SECTION 2: WHAT IS A CACHE HIT / CACHE MISS? (Different from a Write Hit/Miss) 
──────────────────────────────────────────────────────────────────────────────
- 🔥This is INDEPENDENT of whether we want to READ / WRITE.
- 🔥it is ONLY concerned with whether the data requested is
    FOUND IN THE CACHE OR NOT.
- 🔥it DOES NOT concern itself with what we want to do with the data,
    that comes later and is an independent set of action(s).
    - If we want to read to a particular memory address and the data is not in the cache, 
      it triggers a READ to the MM.
    - If we want to write to a  particular memory address and the data is not in cache,
      we have encountered what is called a WRITE MISS. How we deal(diff policies) with these
      kinds of misses is what we'll effectively discuss in a bit. (we discuss about write HITS as well).

- 🔥CACHE HIT: Requested data is found in the cache → Fast access.
- 🔥CACHE MISS: Requested data is NOT in cache → Needs to be fetched from lower level (RAM, SSD, etc).
                Types of misses:
                  - Compulsory miss (first-time access)
                  - Capacity miss (cache too small)
                  - Conflict miss (mapping issue)
- what we want to do with is "not our concern".

──────────────────────────────────────────────────────────────────────────────
SECTION 3: WRITE HIT vs WRITE MISS (IN-DEPTH)
──────────────────────────────────────────────────────────────────────────────
- WRITE HIT: A condition where the processor writes data to a memory address
             that is already present in the cache.

  ◉ WHAT HAPPENS?
    - The cache controller detects that the target memory block (cache line)
      is already cached.
    - Based on the write policy:
        → In **Write-through**, the new value is written to both cache and memory.
        → In **Write-back**, the new value is written only to the cache,
          and the block is marked as **dirty**.

  ◉ IMPLICATIONS:
    - Fast: No need to fetch data from memory.
    - Simplifies timing, but may still trigger coherence protocols in multi-core setups.
    - Requires hardware tracking (e.g., dirty bits in write-back).

  ◉ COHERENCE CONTEXT:
    - In MESI: The block must be in **Modified** or **Exclusive** state for write.
    - If another core has the block in **Shared**, it must be invalidated first.

──────────────────────────────────────────────────────────────────────────────
- WRITE MISS: A condition where the processor writes to a memory address that
              is **not currently present** in the cache.

  ◉ WHAT HAPPENS?
    - The cache controller does **not** find the requested block.
    - Two scenarios based on **miss policy**:
        → **Write-allocate** (aka Fetch-on-write):
             - Cache brings the block from memory into cache.
             - Then performs the write in cache.
        → **No-write-allocate** (aka Write-around):
             - The write goes **directly to main memory**.
             - The block is not brought into the cache.

  ◉ IMPLICATIONS:
    - Slower than a write hit due to memory access on miss.
    - May pollute cache if the block is unlikely to be reused (hence NW-allocate exists).
    - Write-allocate typically paired with write-back for locality.
    - No-write-allocate typically paired with write-through to reduce unnecessary fills.

  ◉ COHERENCE CONTEXT:
    - If cache line is fetched (write-allocate), coherence actions (invalidations) may
      be triggered across other cores.
    - In write-around, coherence is simpler, as no new cache copy is introduced.

SECTION 4: CACHE COHERENCE (IN-DEPTH)
──────────────────────────────────────────────────────────────────────────────
- CACHE COHERENCE: The property that ensures that multiple caches reflect the most recent value of shared data.

  ◉ WHY NEEDED?
    - Multi-core processors may have separate L1/L2 caches.
    - If Core A updates a data value, Core B must not read a stale version from its cache.

  ◉ COHERENCE PROTOCOLS (Examples):
    - MESI: Modified, Exclusive, Shared, Invalid
    - MOESI: Modified, Owned, Exclusive, Shared, Invalid
    - MSI: Simplified version with Modified, Shared, Invalid

  ◉ LINK TO WRITE POLICIES:
    - Write-back caches complicate coherence (data may not be updated in memory).
    - Write-through helps keep memory and caches synchronized (easier for coherence).

──────────────────────────────────────────────────────────────────────────────
SECTION 5: WRITE POLICIES — TYPES & MECHANICS
──────────────────────────────────────────────────────────────────────────────
There are two main orthogonal categories:

  █ CATEGORY 1: MEMORY UPDATE STRATEGY (Write HITS)
  - WRITE-THROUGH (WT): Updates the block in cache AND writes immediately to memory.
  - WRITE-BACK (WB): Updates the block in cache only; memory is updated later,
                     when a cache block is evicted (dirty bit used) OR a cache flush is 
                     requested.
                     In WB approach, multiple changes to a cache line are combined into
                     a single WRITE to main memory reduing the #memoryWrites needed. 
                     Also reduces traffic on the memory bus. ---------> This conserves bandwidth and Increases overall performance.

                                                            

  █ CATEGORY 2: MISS HANDLING STRATEGY
  - WRITE-ALLOCATE (WA): On a write-miss, the block is brought into the cache, then written to.
  - NO-WRITE-ALLOCATE (NWA): On a write-miss, write is sent directly to memory; cache is not updated.

──────────────────────────────────────────────────────────────────────────────
SECTION 6: EXPANDED EXPLANATION OF POLICIES
──────────────────────────────────────────────────────────────────────────────

◉ WRITE-THROUGH (WT) (for WRITE HIT)
---------------------
➤ Cache and memory are updated simultaneously on each write.
➤ Ensures memory always has the latest copy.
➤ Simplifies coherence protocols.
➤ Increases memory traffic.

  ▓ STEPS:
    1. CPU writes data.
    2. Cache checks for presence of block.
    3. If hit: update cache block + write-through to memory.
    4. If miss + WA: bring block into cache, update both.
    5. If miss + NWA: write directly to memory, skip cache.

◉ WRITE-BACK (WB) (for WRITE HIT)
------------------
➤ Writes go only to cache initially; memory updated later (lazy strategy).
➤ Dirty bit is used to track modified blocks. (more complex cache management)
➤ On eviction of dirty blocks, they’re written back to memory.
➤ Reduces memory traffic but increases complexity.
➤ Suffers from the risk of data loss if the cache fails, or power is lost 
  before the modified data is written to main memory.

  ▓ STEPS:
    1. CPU writes data.
    2. Cache checks block:
      ▪ Hit: mark block dirty and update.
      ▪ Miss + WA: fetch block, mark dirty, write.
      ▪ Miss + NWA: write directly to memory.
    3. On eviction:
      ▪ If dirty: write to memory before eviction, ensuring that main memory has the,
                  latest version of the data. 
      ▪ If clean: straight eviction. (since the data is essentially the same in main memory,
                            therefore no write required.)


  🙏NOTE🙏:
  During a cache FLUSH, the Cache controller checks every cache line. 
  If for a line, the Dirty Bit is set to 1,
      The cache immediately writes the data to main memory, and sets the dirty bit to 0(indicating that memory and cache are once again in sync).

SECTION 8: HARDWARE CONSIDERATIONS
──────────────────────────────────────────────────────────────────────────────
- WRITE-BACK needs:
  ▪ Dirty bit per cache block.
  ▪ Replacement policy aware of dirtiness.
  ▪ Write buffers to batch memory writes.

- WRITE-THROUGH needs:
  ▪ Write buffer to hide memory latency.
  ▪ Increases pressure on memory bus.

──────────────────────────────────────────────────────────────────────────────
SECTION 9: IMPACT ON CACHE COHERENCE
──────────────────────────────────────────────────────────────────────────────
- WRITE-THROUGH:
  ▪ Memory always up-to-date.
  ▪ Simplifies inter-core coherence.
  ▪ More memory traffic → may slow down.

- WRITE-BACK:
  ▪ Memory may lag cache.
  ▪ Protocols (like MESI) must track ownership.
  ▪ Complex but efficient.

──────────────────────────────────────────────────────────────────────────────
SECTION 10: WRAP-UP
──────────────────────────────────────────────────────────────────────────────

■ Write operations are more intricate than reads due to data propagation and synchronization.
■ Write-back + write-allocate is most common in modern CPU caches. ==========> Effective in CPU L1 Data cache, as it is 
                                                                               (Optimized for high performace) tasks with 
                                                                               FREQUENT reads/writes on the SAME data.


■ Write-through + no-write-allocate is simpler, (Optimized for Data Consistency)==========> Useful for applications such as Logging Systems,
                                                                                            where data is WRITTEN ONCE, BUT NOT LIKELY to be used in the near future.
                                                                                            Since there is little benefit in caching this type of data, this 
                                                                                            policy combination helps avoid fetching rarely used data into the cache and
                                                                                            prevents cache pollution.  

■ Cache coherence demands play a significant role in selecting the appropriate write strategy.
■ Real systems use hybrid and optimized approaches depending on latency, power, and design constraints.
