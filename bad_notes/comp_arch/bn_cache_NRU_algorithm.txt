
=========================================
# NOT RECENTLY USED (NRU) CACHE ALGORITHM
=========================================


## 1. ALGORITHM OVERVIEW

The Not Recently Used (NRU) algorithm is a simple, low-overhead page replacement policy that, 
approximates LRU behavior while requiring minimal hardware support. It relies on two status bits per page:

- **Referenced (R) bit**: Set when a page is accessed (read or written)
- **Modified (M) bit**: Set when a page is written to (dirty bit)

## 2. OPERATING MECHANISM

### 2.1 Core Operation

1. When the system needs to find a page to evict, it categorizes all pages into four classes:
   - **Class 0**: Not referenced, not modified (R=0, M=0)
   - **Class 1**: Not referenced, modified (R=0, M=1)
   - **Class 2**: Referenced, not modified (R=1, M=0)
   - **Class 3**: Referenced, modified (R=1, M=1)

2. NRU removes a page at random from the lowest-numbered non-empty class.

3. The referenced bit is periodically reset (typically by the OS at regular intervals, 
such as during clock interrupts) to distinguish pages that have been recently used from those that have not.

### 2.2 Eviction Priority Logic

```
If Class 0 has pages:
    Evict random page from Class 0
Else if Class 1 has pages:
    Evict random page from Class 1
Else if Class 2 has pages:
    Evict random page from Class 2
Else:
    Evict random page from Class 3
```

### 2.3 Reset Cycle

The R bits are reset at regular intervals (often 20-30ms). This establishes a time window that defines "recently used." 
Pages accessed within this window will have R=1, while others will have R=0.

## 3. IMPLEMENTATION DETAILS

### 3.1 Data Structures

- Page table entries with R and M bits
- Optional: Four lists tracking pages in each class (for efficient selection)
- Optional: Clock interrupt handler to reset R bits

### 3.2 Pseudo-implementation

```
// Called when a page fault occurs and a page must be evicted
function select_victim_page():
    // Build the four classes
    class0 = class1 = class2 = class3 = empty_list
    
    for each page in memory:
        if page.R == 0 and page.M == 0:
            add page to class0
        else if page.R == 0 and page.M == 1:
            add page to class1
        else if page.R == 1 and page.M == 0:
            add page to class2
        else: // page.R == 1 and page.M == 1
            add page to class3
    
    // Select victim from lowest non-empty class
    if not class0.empty():
        return random_page_from(class0)
    else if not class1.empty():
        return random_page_from(class1)
    else if not class2.empty():
        return random_page_from(class2)
    else:
        return random_page_from(class3)

// Called periodically (e.g., by a timer interrupt)
function reset_reference_bits():
    for each page in memory:
        page.R = 0
```

## 4. ADVANTAGES AND LIMITATIONS

### 4.1 Advantages

- **Low implementation overhead**: Requires only two bits per page
- **Hardware-friendly**: Easy to implement in hardware
- **Good approximation**: Performs reasonably well compared to more complex algorithms
- **Favors unmodified pages**: Reduces expensive write-backs to storage

### 4.2 Limitations

- **Coarse granularity**: Cannot distinguish between degrees of "recentness" within a cycle
- **Random selection**: Potentially suboptimal choices within classes
- **Clock synchronization**: Performance depends on reset interval tuning
- **Cold-start issues**: New pages immediately marked as referenced

## 5. COMMON USAGE CONTEXTS

### 5.1 Operating Systems

- **Virtual Memory Systems**:
  - Many early UNIX variants
  - Early versions of Windows (before more sophisticated algorithms)
  - Some lightweight or embedded OS environments

### 5.2 CPU Caches

NRU is rarely used in modern CPU caches for several reasons:
- CPU caches typically use set-associative mapping requiring more specialized algorithms
- L1/L2 caches need faster decisions than NRU can provide
- Modern CPUs use variations of PLRU (Pseudo-LRU) or RRIP (Re-Reference Interval Prediction)

However, elements of NRU philosophy appear in:
- **L3/Last-level caches**: Some designs use NRU-like bits as part of hybrid schemes
- **Intel's Core i-series**: Uses a variant of NRU logic in some cache designs
- **ARM processors**: Some implementations use referenced bits in their cache replacement logic

### 5.3 Other Applications

- **Database Buffer Pools**: Some older DBMS systems
- **Embedded Systems**: Memory-constrained environments where simplicity is valued
- **Network Equipment**: Router/switch caches where low overhead is critical
- **Virtual Machine Hypervisors**: For managing guest memory pages

## 6. MODERN VARIATIONS

### 6.1 Enhanced NRU (ENRU)

- Uses multiple history bits to track references across several time periods
- Provides finer granularity of "recentness"

### 6.2 Clock Algorithm

- Conceptually similar to NRU but uses a circular buffer and pointer
- Often called "Second-Chance" algorithm
- More efficient implementation avoiding full memory scans

### 6.3 RRIP (Re-Reference Interval Prediction)

- Modern algorithm that extends NRU concepts
- Uses multi-bit counters instead of single reference bits
- Dynamic prediction of when pages will be referenced again

## 7. PRACTICAL CONSIDERATIONS

- **Reset Interval**: Too frequent resets reduce the algorithm's effectiveness; too infrequent make it less responsive
- **Random Selection**: Can be improved by using round-robin or other deterministic methods within classes
- **Hardware Support**: Most effective when R bits are automatically set by hardware on access
- **Memory-Mapped Files**: Special handling may be needed for shared pages

## 8. CONCLUSION

The Not Recently Used algorithm represents a practical compromise between implementation simplicity and performance. 
While rarely used in its pure form in modern high-performance systems, its concepts form the foundation 
for many contemporary replacement policies. Its primary strengths lie in its minimal hardware requirements, 
  and reasonable approximation of more complex algorithms, making it suitable for environments where 
  resource constraints outweigh the need for optimal replacement decisions.
